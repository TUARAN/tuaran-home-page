# Web LLM 集成对照记录

## 1. 背景

当前仓库的 `/web-llm` 页面不是从零设计，而是参考本地独立项目 `/Users/tuaran/Documents/GitHub/webllm` 迁移过来的站内集成版。

这份文档记录两件事：

- 当前站点应该如何理解这套能力的核心思路
- 对照本地 `webllm` 项目后，哪些实现可以直接继承，哪些地方不能机械照搬

## 2. 本地 `webllm` 参考项目是什么

本地参考项目是一个独立的浏览器端 WebGPU LLM demo，技术形态比较纯：

- 构建体系：`Vite`
- 页面入口：`index.html + src/main.js`
- 推理依赖：`@huggingface/transformers`
- 模型形态：`onnx-community/Qwen3.5-*-ONNX`
- 运行设备：`webgpu`
- 能力范围：
  - 模型选择
  - 点击后手动加载模型
  - 文本对话
  - 图片上传
  - IndexedDB 会话历史
  - 流式输出
  - 速度统计
  - 运行环境诊断

它的定位是“把浏览器本地跑模型这件事先跑通”，重点是能力闭环，不是站点集成。

## 3. 当前 `tuaran-home-page` 中的 `/web-llm` 是什么

当前仓库里的 `/web-llm` 不是独立 demo，而是把上面那套能力嵌入一个现有 `Next.js 15 App Router` 网站里。

因此它的目标不是完整复刻参考项目页面，而是做两层转换：

1. 能力迁移
把 `webllm` 里已经验证过的浏览器端推理链路迁进来。

2. 站点融合
让这套能力服从当前站点的页面结构、状态组织、样式体系和部署方式。

换句话说，参考项目解决的是“浏览器里能不能跑起来”，当前项目解决的是“如何把这套能力变成站内页面能力”。

## 4. 当前项目的核心思路

这部分是后续维护时最重要的判断标准。

### 4.1 这是浏览器本地推理，不是服务端 AI API

`/web-llm` 的核心前提：

- 不依赖服务端 LLM API
- 不走 OpenAI、Claude 或自建推理服务
- 模型在用户浏览器里下载、缓存和运行
- 推理设备直接指定为 `webgpu`

因此这个页面的问题排查顺序，永远应优先看：

- 浏览器是否暴露 `navigator.gpu`
- 当前页面是否是安全上下文
- COOP / COEP 响应头是否正确
- 模型资源能否正常下载

而不是先去查服务端接口。

### 4.2 页面必须手动加载模型，不能首屏自动拉起

这是参考项目和当前项目都应该保留的原则：

- 默认进入页面时不自动加载模型
- 用户先选模型，再显式点击“加载模型”
- 加载完成前，不允许发送消息

原因很简单：

- 模型体积大
- 首次下载慢
- 不同设备显存差异大
- 自动加载会让站点页面失控

`/web-llm` 不是普通聊天框，它本质上是一个需要显式初始化的浏览器运行时。

### 4.3 环境诊断要可见，但不要过度阻塞

这次排查最关键的结论就在这里。

参考项目的策略是：

- 检查 `WebGPU`
- 检查 `window.isSecureContext`
- 检查 `crossOriginIsolated`
- 页面上明确显示这些状态
- 如果加载失败，把环境诊断拼进错误信息里

但参考项目并没有把 `crossOriginIsolated` 当成按钮能否点击的硬前提。

它的实际语义是：

- `WebGPU` 不可用：无法在 GPU 上加载模型
- 非安全上下文：浏览器能力受限，通常也不该继续
- `crossOriginIsolated` 未启用：某些 ONNX / WASM 优化可能不可用，但不应该直接阻止手动尝试

因此当前项目的正确策略应该是：

- `hasWebGPU && isSecureContext` 才算“允许尝试加载模型”
- `crossOriginIsolated` 作为运行状态和兼容性提示
- 加载失败时仍然把 `crossOriginIsolated` 暴露给用户排查

这就是这次修复的核心判断。

### 4.4 当前项目不是复刻 Vite demo，而是做 Next.js 集成

这意味着实现时应优先遵守当前仓库结构，而不是照抄参考项目文件组织。

当前合理分层是：

- `app/web-llm/page.jsx`
  页面壳
- `app/web-llm/WebLlmPageClient.jsx`
  页面交互、按钮状态、聊天 UI、会话切换
- `app/web-llm/lib/runtime.js`
  模型加载、推理、流式更新、环境快照
- `app/web-llm/lib/sessionStore.js`
  IndexedDB 本地持久化
- `app/web-llm/lib/siteContext.js`
  给模型注入站点背景知识

判断标准应是“能力分层是否清楚”，而不是“是否长得和 `src/main.js` 一样”。

### 4.5 当前项目比参考项目多了一层“站点问答上下文”

这是当前集成版相对参考 demo 的一个关键增强。

参考 `webllm` 的重点是通用本地聊天。

当前项目还额外做了：

- 读取站点上下文材料
- 将站内信息组织进 prompt / conversation
- 让模型回答更偏向“tuaran 站内内容问答”

所以这里不是纯通用聊天器，而是“站点上下文增强的本地模型问答页”。

## 5. 本地 `webllm` 与当前项目的主要对照

### 5.1 相同点

- 都使用 `@huggingface/transformers`
- 都使用 `onnx-community/Qwen3.5-*-ONNX`
- 都要求用户手动点击加载模型
- 都有运行环境诊断
- 都有流式输出
- 都有 IndexedDB 本地会话
- 都把图片输入视为可选增强

### 5.2 不同点

- 参考项目是 `Vite` 单页 demo；当前项目是 `Next.js App Router` 页面集成
- 参考项目追求能力验证；当前项目还要兼容站点布局、主题和部署
- 参考项目的 COOP / COEP 在 `vite.config.js` 里配；当前项目在 `next.config.js` 里配
- 参考项目把环境诊断更多当成提示；当前项目更容易误把它们写成 UI 硬门槛
- 当前项目多了站点上下文注入，不是纯空白聊天

### 5.3 这次实际踩到的问题

当前项目之前出现“加载模型按钮点击不了”，根因不是浏览器完全不支持，而是集成时把条件收得过死：

- 按钮禁用条件写成了 `hasWebGPU && isSecureContext && crossOriginIsolated`
- 运行时预检也要求 `crossOriginIsolated === true`

这和参考项目不一致。

参考项目的行为更合理：

- 允许用户在基础环境满足时点击尝试
- 如果失败，再结合环境诊断反馈问题

所以当前项目后续如果继续演进 `/web-llm`，应优先保持这个原则，避免再次把“诊断项”误写成“绝对阻塞项”。

## 6. 当前推荐保留的实现原则

后续无论谁再改 `/web-llm`，都建议遵守下面这些原则：

1. 不自动加载模型
始终保留“用户显式点击后才初始化模型”。

2. 环境状态提前显示
在模型加载前就显示 `WebGPU / secure context / cross-origin isolation`。

3. 只把必要条件做成硬门槛
当前建议的硬门槛是：

- `navigator.gpu` 可用
- `window.isSecureContext === true`

4. 把 `crossOriginIsolated` 视为兼容性信号
它很重要，但默认不应直接禁用“加载模型”按钮。

5. 错误提示必须带诊断信息
加载失败和推理失败都要让用户看得到原因，不要静默失败。

6. 保持站点级分层
不要把页面 UI、会话存储、模型运行时重新塞回一个超大文件。

7. 优先维护站点问答体验
当前项目不是纯 demo，站点上下文注入能力应继续保留。

## 7. 与头部配置相关的结论

当前项目的 `/web-llm` 路由依赖 COOP / COEP 头。

对照本地 `webllm` 项目后，当前仓库建议保持：

- `Cross-Origin-Embedder-Policy: credentialless`
- `Cross-Origin-Opener-Policy: same-origin`

并确保头覆盖整个 `/web-llm/:path*` 路由段，而不是只写单一路径。

原因不是“为了形式一致”，而是为了减少浏览器侧 WebGPU / ONNX 运行时的兼容性问题。

## 8. 结论

以后理解这个功能时，应该始终记住一句话：

`/web-llm` 的本质不是一个普通聊天按钮，而是一套嵌入站点页面中的浏览器本地推理运行时。

因此维护重点应放在：

- 运行环境诊断是否准确
- 模型加载链路是否清楚
- 按钮禁用条件是否合理
- 站点上下文注入是否仍然成立
- Next.js 集成是否保持分层清晰

如果后续再参考本地 `webllm` 项目，不要机械照抄页面，而要优先对齐它的核心运行原则。
