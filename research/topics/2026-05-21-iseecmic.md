> **本篇定位**：把本地 `iseecmic`（“我眼中的 CMIC”）项目的观察内容，整理为一份调研。原项目是一个把分散 Excel 数据重构为“可解释网站”的结构化观察系统；本篇抽取其核心命题、框架与洞察。**已加密，仅自己查看。**

## 一、是什么：核心命题

一句话：**公司不是组织架构图，而是一个由流程、任务、项目、奖项、预置模板、人与推动关系构成的系统。**

iseecmic 刻意不做“企业官网”，而是做一张**内部观察地图**——它不解释“公司应该是什么样”，而是观察“公司实际怎么运转、靠什么维持秩序”。

## 二、观察框架：把公司拆成可解释的实体

项目把模糊的“公司”拆成 7 类实体 + 一组关系，使观察可被结构化：

| 实体 | 含义 |
|---|---|
| Project 项目 | 组织行为的核心观察单位 |
| Department 部门 | 项目与角色的归口 |
| Role 角色 | 抽象的组织行为角色（非真实个人）|
| Artifact 成果物 | 奖项、专利、政策、流程等项目衍生物 |
| Observation 观察 | 由多类证据拼接出的结论 |
| TimelineEvent 时间线事件 | 认知阶段 |
| RawSheet 原始表 | 结构化实体的 Excel 来源 |

关系骨架：部门 1→N 项目 / 部门 1→N 角色 / 项目 1→N 成果物 / 项目 N↔N 角色 / 原始表 1→N 项目与成果物。

**关键设计**：用“角色”代替真实个人——观察组织行为，而不是评价具体的人。

## 三、样本：三部门 × 三项目

| 部门 | 类型 | 定位 | 样本项目 | 申报奖励 |
|---|---|---|---|---|
| 创新产品部 | 业务探索 | 把概念转化为项目名册与可展示能力 | AI 能力开放平台 | ¥30 万 |
| 数据智能部 | 能力支撑 | 把数据治理转化为口径、台账、平台 | 数据资产治理专项 | ¥12 万 |
| 综合管理部 | 组织运行 | 把分散工作沉淀为流程与申报材料 | 成果申报支撑台账 | ¥5 万 |

三个项目分别对应三种典型形态：**平台型**（包装分散能力为统一能力池）、**治理型**（建一个可反复引用的基础面）、**台账型**（让成果材料随时可被调度）。

## 四、组织角色画像（六型）

项目用六种行为角色描述“人在组织里做什么”，而非介绍真实个人：

| 角色 | 定义 | 组织价值 |
|---|---|---|
| 推动型 | 把模糊目标变成明确动作 | 保证项目不停在口头 |
| 汇总型 | 把零散进展整理成表格台账 | 让组织看见正在发生什么 |
| 协调型 | 在部门间找最小共识 | 降低跨部门摩擦 |
| 填表型 | 把临时要求转成可提交字段 | 维持输入输出的完整性 |
| 系统型 | 把重复动作固化成流程接口 | 降低重复劳动 |
| 救火型 | 临期补齐缺失项 | 高压节点维持交付 |

## 五、运行逻辑：三条核心洞察

这是 iseecmic 最锋利的部分——解释“表格、空白、优先级”背后的系统行为。

**1. 为什么公司不断创建表格**

> 表格是把抽象要求转译成可分派字段的最低成本接口。

- 管理目标往往是抽象的，但组织执行需要具体字段。
- 表格把责任拆成列、把协作拆成填写动作。
- 字段一旦形成，汇总、追责、比较、复用都变得容易。
- 所以表格不是附属物，而是组织把复杂现实压缩为可治理对象的手段。

**2. 为什么很多模板处于待填状态**

> 待填 = 组织先把容器建好，再等任务 / 资源 / 口径来填充。

- 模板常先于真实内容出现——制度设计往往先于稳定业务。
- 预留字段保留了未来接入的可能，也说明任务尚未稳定。
- 有些表存在的意义是“证明流程已部署”，而非“证明内容已成熟”。
- 所以待填模板本身就是一种状态信息：系统在预留接口。

**3. 什么东西会被优先填满**

> 与考核、申报、预算、领导关注直接相关的字段，最先获得内容。

- 能直接进入汇报链条的内容，优先级最高。
- 能形成奖项 / 案例 / 预算依据的项目，更容易被补齐材料。
- 责任明确、联系人清晰、时间节点刚性的表，最容易被快速填满。
- 结论：注意力分配遵循**可见性与可转译性**，而非工作难度。

## 六、价值的双面性

项目用 `buildProjectAnalysis()` 给每个项目自动生成三个分析字段，把“看上去”与“实际上”分开：

- **看上去的价值**：取决于标签热度、奖励信号、责任出口是否清晰——即“概念够不够先进、能否绑定申报与预算、有没有明确接口人”。
- **实际承担的功能**：取决于描述里的“平台 / 中台 / 试点 / 专项 / 台账”等关键词——多数项目实际更像“基础设施接口”或“维持任务闭环的记录单元”，并承担“把工作转译为可统计成果”的作用。
- **组织上为什么存在**：它能把抽象战略翻译成**可分派、可汇总、可申报**的对象；同时为部门提供稳定的存在感与资源叙事；即便没有奖金，也能沉淀为下一轮表格与汇报的依据。

一句话：**一个项目“看上去的价值”和“它在系统里真正承担的功能”，常常是两回事。**

## 七、认知演进

观察者视角的四个阶段：

| 阶段 | 看到了什么 |
|---|---|
| 初期 | 看到任务——一个个被分派的事项 |
| 中期 | 看到流程——任务被模板、节点、审批串联 |
| 后期 | 看到系统结构——项目 / 表格 / 奖项 / 部门相互支撑 |
| 最终 | 看到组织运行逻辑——公司靠“可汇总对象”维持秩序、分配注意力、证明自身运行 |

## 八、原始档案

观察可回溯到 Excel 源头，4 张代表性表：

| 表 | 类型 | 状态 | 行数 | 说明 |
|---|---|---|---|---|
| 项目清单 | 项目 | 部分填写 | 48 | 字段完整但口径不统一 |
| 奖项与成果申报 | 奖项 | 高频更新 | 26 | 与季度汇报、年度评优直接相关，填充度高 |
| 政策制度跟踪 | 政策 | 低频更新 | 14 | 固化运行要求，文字型字段较多 |
| 创新储备项目模板 | 预置模板 | 待启用 | 0 | 模板已存在，但项目池尚未形成稳定输入 |

注意：状态为“高频更新”的，恰是与评优申报绑定的表——印证第五节的优先级逻辑。

## 九、个人结论

- **一句话定性**：iseecmic 不是一个企业站，而是一套“把公司当系统来读”的观察方法——核心价值在于**把组织的隐性运行逻辑显性化**。
- **最有价值的部分**：第五节三条运行逻辑 + 第六节价值的双面性。它们解释了一个常见困惑——“为什么忙的不一定是最重要的，最重要的反而没人填”。
- **可延伸**：项目预留了 MCP / Agent 接口（`sourceSheetIds`、`confidence`、`originWorkbook`），后续可把新增 Excel 自动转成 Observation。
- **边界**：样本只有 3 部门 3 项目，是模型骨架而非全量画像；角色是抽象类型，不对应具体个人。
